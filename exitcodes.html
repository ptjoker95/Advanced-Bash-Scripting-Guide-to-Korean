Appendix E. Exit Codes With Special Meanings

Table E-1. 지정된 종료 코드들

종료 코드 숫자 의미 예 주석
1 일반적인 에러들 let "var1 = 1/0" "0 으로 나누기" 와 다른 허용되지 않는 동작들과 같은, 잡다한 에러들
2 쉘 내부 명령어의 잘못된 사용 (Bash 문서에 따름) empty_function() {} 키워드가 빠지 거나 명령어, 혹은 파일접근권한 문제 (그리고 잘못된 바이너리 파일 비교에 쓰인 diff 의 리턴 코드).
126 요청된 명령어를 실행할 수 없음 /dev/null 파일접근권한 문제 혹은 명령어를 실행할 수 없음
127 "명령어를 찾을 수 없음" illegal_command 경로 $PATH 문제거나 오타
128 exit 에 잘못된 인수가 쓰임 exit 3.14159 exit 에는 0 - 255 범위의 정수만 사용할 수 있음 (첫번째 각주를 참고)
128+n "n" 의 심각한 에러 kill -9 $PPID of script $? 가 137 (128 + 9) 을 리턴함
130 스크립트가 Control-C 로 종료됨 Ctl-C Control-C 는 2 의 심각한 에러 시그널을 가짐, (130 = 128 + 2, 위 참조)
255* 종료 상태가 범위를 벗어남 exit -1 exit 는 0 - 255 범위 내의 정수만을 인수로 가질 수 있음

위의 표에 따르면, 1 - 2, 126 - 165, and 255 [1] 종료 코드는 특별한 뜻을 가지고 있고, 사용자가 지정한 종료 매개변수로 사용해서는 안됩니다. 종료 코드 127 를 가진 스크립트는 문제가 생겼을 때 분명 혼동을 일으킬 것입니다 ( "명령어를 찾을 수 없음" 인지? 사용자가 지정한 것인지?). 하지만, 많은 스크립트들은 일반적인 에러를 위해 종료 코드 1 을 사용합니다. 종료 코드 1 은 너무 많은 에러를 의미하기 때문에, 디버깅에 그다지 유용하지는 않습니다.

종료 상태 숫자를 시스템화하려는 시도가 있었지만 ( /usr/include/sysexits.h 를 참고하십시요), 이는 C, C++ 프로그래머들을 위한 것이었습니다. 스크립트를 위한 비슷한 표준이 필요할 수도 있을 것입니다. 이 문서의 제작자는 사용자 지정 종료 코드는 64 - 113 로 제한하고 (성공시를 위한 0 를 포함해서), C/C++ 표준을 준수하는 것을 제안합니다. 이는 50개의 유용한 코드들을 할당하고, 문제가 생긴 스크립트들을 보다 직접적으로 다룰 것입니다. [2] 예제 9-2 오버라이딩 환경 벗어나기를 제외하고, 이 문서에 있는 예제들에 있는 사용자 정의 종료 코드들은 이 표준을 따르고 있습니다.

쉘 스크립트가 종료된 뒤 명령어 줄에서 $? 로 나오는 결과는 위의 표에서 설명한 것과 일치하지만 이것은 Bash 나 sh 에서만 적용됩니다. C-shell 이나 tcsh 의 경우에는 몇몇 경우에 다른 결과값이 나옵니다.

Notes

[1]

종료값의 범위를 벗어나는 값은 기대하지 않은 종료 코드가 반환될 수 있습니다. 255 보다 큰 종료값은 종료 코드로 나머지 를 반환합니다. 예를 들어, exit 3809 는 종료 코드로 225 (3809 % 256 = 225) 를 반환합니다.

[2]

An update of /usr/include/sysexits.h allocates previously unused exit codes from 64 - 78. It may be anticipated that the range of unallotted exit codes will be further restricted in the future. The author of this document will not do fixups on the scripting examples to conform to the changing standard. This should not cause any problems, since there is no overlap or conflict in usage of exit codes between compiled C/C++ binaries and shell scripts.
